시스템 프로그래밍 ↔ 운영체제

시스템 프로그래밍은 운영체제를 공부하는 것과 같다.

프로세스 부분은 운영체제와 아키텍처에 대한 이해가 필요하다

운영체제 분야만큼은 하드웨어를 강제로 배워야한다.

시스템 관리 또한 하드웨어를 배워야한다.

프로세스부분은 운영체제, 컴퓨터 구조, c 언어가 합쳐진 부분이다.

운영체제를 배우기 전까지는 이해하기 어려울 수 있다.

프로세스와 프로그램의 구분***

스토리지에 저장되어 있는 실행파일을 프로그램이라고한다. 실행중이지 않은 상태

프로세스는 실행할 수 있는 모든 자원(메모리)을 할당받은 상태로 실행중인 상태 이다.

운영체제 프로그램을 메모리에 적재되고 프로세스로 다룸

프로세스는 서로 독립적인 메모리 공간을 가짐. 다른 프로세스 영역에 접근 시  접근한 프로세스를 강제 종료한다.

영역을 정확히 알아야한다.

운영체제는 프로세스마다 고유한 번호를 할당한다.

프로세스의 모든 정보는 커널에 의해 관리

모든 프로세스는 항상 실행중이지 않고 실행-대기-잠자기(오랜시간 대기, sleep)-대기-실행-종료 등의 생명 주기를 가진다.

디스크 →메모리 적재 2가지 방식 커맨드라인과 gui(클릭) gui은 항상 같은 형태로 실행, 커맨드라인 실행은 ls-c,ls-g 같이 형태 변경가능

프로세스 관리 pid(프로세서 번호)로 관리되는 것이 가장 좋다. 커널영역에 프로세스 테이블을 만들어 관리

윈도우 → 작업관리자

linux → ps -ax (bsd 계열과 sys v이 스타일 다르다. 시스템 v는 ef)

mac은 bsd, linux는 sys v 감안해서 사용해야한다.

운영체제는 프로세스가 관심이 없기 때문에 프로그램 실행마다 독립된 프로세스로 생성된다. (다중 인스턴스)

하나의 프로그램(실행 파일)로 부터 나온 다중 인스턴스라고 부른다.

pid(프로세스 번호)가 같은 프로세스는 존재할 수 없다.(존재시 커널이 죽음 블루스크린)

현대 기준

cpu 주소 공간(물리 주소 공간) : cpu가 전기 신호를 통해 엑세스하는 공간

32bit - 64pins(주소선) → 2^32개의 주소 → 2^32 바이트 → 4gb

64bit - 128pins(주소선) → 2^64개의 주소 → 2^64 바이트 → —

공간 크기는 주소선의 수에 의해 결정

Cpu 공간보다 큰 메모리? 있어도 엑세스 불가능 ***

Cpu가 설치된 메모리의 주소 영역을 넘어 액세스하면 시스템 오류

32bit에서 4기가 이상의 메모리 용량은 페이즈는 늘려 램디스크(지정할 수 있는 크기가 정해저 있다.)로 사용해 엑세스 할 수 있다.

프로세스가 존재하는 공간은 가상공간

임베디드 시스템에서는 메인 메모리의 크기를 항상 염두하고 프로그래밍해야한다.

메인 메모리에 적재하려다가 보니 더 관리하기 쉽게하기위해 프로그램의 데이터를 용도에 따라 구분하였다.

1. 코드 영역(기계어만)
2. 데이터 영역(전역변수만)
3. 힙 영역(동적 할당)
4. 스택 영역(지역변수 포함)

- 실제로 공간을 차지 하고 있는 것이 아닌 뭉쳐있다고 생각하는것.
- 1234순? 연속적으로 존재하고 있다.

위와 같은 가상 공간을 프로세스 주소 공간이라고한다.

실행중에 접근할 수 있는 공간이기에 cpu가 액세스 할 수 있는 전체 크기를 넘어설  수 없다.

프로세스 주소 공간은 프로그램 혼자가 사용한다고 생각한다. → 물리주소 공간위에 가상 공간인 프로세스 주소 공간을 사용하기에 하나의 프로세스가 4GB 전부를 액세스할 수 있다. ****

커널을 통하여 공간을 사용해야 하기에 프로세스는 4GB에서 사용자 공간 + 커널 공간을 나누어 사용된다.  (꺼꾸로 자람, 번지로 자람은 나중에 운영체제에서 배울것)

이러한 형태 때문에 프로세스 주소 공간을 가상 주소 공간이라고 한다.

물리 주소 공간을 중심으로 프로그래밍을 하는 것을 iot나 임베디드 시스템 이다.

간단하기 얘기하면 프로세스의 수가 변화하는 시스템에서는 영역을 변화할때마다 할당할 수 없기에 가상 주소를 쓰면 된다. 프로세스의 수가 정해진 것은 물리 주소 공간
